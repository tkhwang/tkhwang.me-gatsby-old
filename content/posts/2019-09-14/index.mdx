여러가지 프로그래밍의 좋은 명언들을 모아둔 [프로그래밍의 정석](https://www.aladin.co.kr/shop/wproduct.aspx?ItemId=120459446) 책 내용 중에서 Chapter2. 원칙: 프로그래밍 가이드라인 내용을 요약해본다. 앞으로 프로그래밍을 하면서 헌법과도 같이 중요한 원칙으로 삼고자 한다.

## 2.1 KISS (Kieep It Simple, Stupid / Keep It Short and Simple)

### (WHY) 코드는 무질서로 향한다.

- 코드를 작성할 때는 최우선 가치를 **단순성**과 **간결성**에 둔다.
- 단순한 코드는 읽기 쉽고 이해하기 쉬우며 수정이 용이하다.

### (HOW) 코드에 불필요한 것을 하지 않는다.

- 단순함을 프로그래밍의 나침반으로 삼자.
- 코드에서 불필요한 것과 넘치는 것을 가능한 한 제거해야 한다.
- 그렇게 하려면 프로그래밍 중에도 코드가 동작할 수 있는 가장 간단한 방밥은 무엇인지 항상 질문을 던저야 한다.
- 코드는 두뇌의 명석함을 뽐내기 위한 것이 아니라 **사용자에게 가치를 제공하기 위한 것**이다. **뽐내려 해서는 안 된다.**
- 지금 필요하지 않다면 지금 작성할 것이 아니다. 필요해졌을 떄 필요한 만큼만 코드를 작성하도록 해서 단순함을 유지하자.

## 2.2 DRY (Don't Repeat Yourself)

### (WHAT) 코드 복사는 금물

- 똑같은 코드를 중복해서 작성해서는 안 된다.
- 코드 중복이 가장 많은 사례는 로직 한 덩어리를 다른 부분에 손쉽게 복사 후 붙여넣기해서 사용했을 때다.
- 정수 literal을 직접 코드에 내장하는 것도 코드 중복에 해당한다. 같은 의미의 정수가 여러 군데에서 사용된다면, 정수가 가리키는 정보가 중복해서 존재하기 떄문이다.

```javascript
function A() {
  if ( value < 16 ) {}
  ...
  if ( value < 16 ) {}
  ...
}
```

- 그저 단순히 코드를 그대로 설명하는 주석도 중복에 해당한다. 코드를 모국어로 번역하기만 한 주석 역시 중복이다.

### (HOW) 코드를 추상화한다.

- **코드를 추상화**함으로써 **중복을 제거**하자.
- 코드 로직을 추상화하려면 처리하는 코드를 묶고 이름을 붙여 함수화, 모듈화한다. 데이터라면 이름을 붙여 상수를 정의한다. 이렇게 이름을 붙인 코드를 반복해서 나오는 부분에 치환한다.

### (관련4) 레거시 코드

- 예전에는 레거시 코드라고 하면 '옛날에 만들어진 이해할 수 없고 변경하기 어려운 코드'를 조롱하는 속어였다.
- 그러나, 최근에는 **테스트에 의한 품질 보호를 중요시할 목적으로 '테스트가 없는 코드'**라고 재정의 되었다.
- 테스트가 없는 코드는 예외 없이 나쁜 코드다. 코드가 깔끔하고 제대로 구조화되어 있어도 테스트가 없다면 레거시 코드다.
- 레거시 코드를 상대하게 되었다면 **일단 테스트부터 수행**하자. 우아하지 않아도 상관없다. 어떤 방법을 사용해서라도 우선 테스트를 수행하고 나서 수정해야 한다. 품질을 지키는데 있어 반드시 필요한 절차이다.

## 2.3 YAGNI (You Ain't Going to Need It.)

### (WHAT) 코드는 필요할 떄 최소한으로

- 코드는 정말로 필요해졌을 때 필요한 코드만을 작성한다는 방침으로 프로그래밍에 임해야 한다.
- 앞으로의 일은 딱 잘라내고 지금 필요한 코드만을 작성하도록 한다.

### (HOW) 코드는 지금 필요한 것만

- 범용성보다는 단순성을 생각하자.

## 2.4 의도를 표현해서 프로그래밍하라 PIE (Program Intently and Expresively)

### (WHAT) 코드의 의도를 전한다

- 코드를 작성할 때는 의도를 명확하게 표현해야 한다.
- 코드는 컴파일러가 아닌 사람이 읽기 위한 것이기 떄문이다.

### (WHY) 코드가 유일한 실마리

- 코드는 소프트웨어 동작을 정확하고 완벽하게 알기 위한 유일한 실마리다.

### (HOW) 코드는 읽기 쉬운 것이 최우선

- 코드를 작성할 때는 '작성하기 쉬움'보다 '읽기 쉬움'을 중시하자.
- 코드는 작성하는 경우보다 읽히는 경우가 훨씬 많은 법이다.
- 코드는 **실행 효율보다 읽는 효율이 우선시** 된다.

### (발전2) 주석을 작성한다

- 주석없이도 읽을 수 있을 법한 이해하기 쉬운 코드를 작성하는 것이 이상적이다.
- 코드는 어디까지나 `what`과 `how`, 즉 `무엇을 하는지`와 `어떻게 하는지` 밖에 표현하지 못한다.
- `why` 즉, `어째서 그것을 하는지`를 표현하려면 주석을 사용할 필요가 있다.

## 2.5 추상화 수준의 통일 SLAP (Single Level of Abstraction Principle)

### (WAHT) 코드 수준을 맞춘다.

- 코드를 작성할 때 높은 수준의 추상화 개념 (고수준)과 낮은 수준의 추상화 개념(저수준)을 **분리**하도록 한다.
- 추상화 단계를 분리할 때는 상하 2계층이 아니라 기능의 복잡도에 따라 여러 계층으로 분리한다.
- 각각의 계층에서는 자신의 계층에 속하는 요소에 대해 추상화 수준을 일치시킨다.
- **함수**는 **추상화 수준에 따라 분할**하고, **같은 함수에 속하는 코드의 추상화 수준은 모두 통일**한다.

### (WHY) 코드에 요약성과 열람성을 가져다 준다

- 코드가 수준이 일치된 함수를 분할되어 있으면 **요약성**과 **열람성**을 동시에 충족한다.

### (HOW) 함수를 구조화 한다

- **함수를 구조화**하자.
- 의도가 전달되기 쉽게끔 처리는 **추상화 수준을 일치시킨 작은 단계의 함수로 변환**하자.
- 함수가 구조화하면 **각 함수**는 **자신보다 한 단계 낮은 수준의 함수를 호출하는 처리가 중심**이 된다. 이처럼 다른 함수를 호출하는 코드로 구성된 함수를 **복합 함수 composed method**라고 부른다.
- 복합 함수는 **최대한 작게** 만든다. 이름으로 의도를 전달하기 위해서라면 처리가 한 라인일지언저 함수로 만들어도 상관없다.
- 복합 함수 안에서는 추상화 수준이 다른 함수를 호출하지 않도록 한다.

### (발전1.) SLAP의 적용 범위

- SLAP은 함수 뿐만 아니라 모듈 등에도 적용된다.
- 추상 클래스 : 높은 수준의 개념
- 해당 상속 클래스 : 낮은 수준의 개념

## 2.6 개방-폐쇄의 원칙 OCP (Open-Closed Principle)

### (WHAT) 코드의 변경은 파급시키지 않는다.

- 코드는 **확장**에 대해서 **열려 있고**, **수정**에 대해서 **닫혀 있는** 2가지 속성을 동시에 충족하도록 설계한다.
- 확장에 대해서 열려 있다 : 코드의 동작을 확장할 수 있다.
- 수정에 대해서 닫혀 있다 : 코드의 동작을 확장하더라도 그 밖의 코드는 전혀 영향을 받지 않는다.

### (HOW) 코드에 인터페이스를 사용한다.

- **인터페이스를 사용**해서 **설계**하자.
- 특정 기능을 가진 모듈을 설계할 때, 모듈의 사용자인 클라이언트가 모듈의 제공자인 서버를 직접 호출하는 방식은 경직된 설계다.
- 클라이언트와 서버 사이에 모듈 상용자를 위한 **클라이언트 인터페이스**를 만들고, 서버는 클라이언트 인터페이스를 구현한다.

## 2.7 Naming is important 명명이 중요하다

### (WHAT) 코드에서 명명은 가장 중요한 과제

- 프로그래밍에서 명명은 가장 중요한 과제로 인식하고 신준하게 착수하도록 하자.
- '이름을 붙이는 행위'와 그런 과정을 통해 생겨난 '이름 자체'에 모두 중요한 가지가 있다.

### (WHY) 코드를 읽는 사람에 대한 사용자 인터페이스

- 이름은 코드를 읽는 사람에 대한 사용자 인터페이스 (UI)다.
- 코드를 읽고 잘 이해한 다음 수정하거나 기능을 추가하려는 것이 목적이다.

### (HOW) 코드는 우선 이름을 정한다

- 프로그래밍은 우선 이름부터 시작하자.
- 항상 사용할 사람과 읽은 사람의 관점에서 명명해주기 바란다.
